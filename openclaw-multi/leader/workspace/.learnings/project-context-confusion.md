# 错误记录：多项目并行时的上下文混淆（2026-03-13）

## 错误描述
在 Project 3 进行中，两次错误地将 Project 2 的内容串到回复中。

### 第一次错误（13:33）
**错误内容**：
- 提到 "Introduction v2"（Project 2 的版本号）
- 提到 "Editor 已整合到 paper.tex"（Project 2 的操作）
- 提到 "Reviewer 审查"（Project 2 的阶段）

**实际情况**：
- Project 3 的 Writer 刚完成 4/6 个 section
- 还没有进入 Editor 或 Reviewer 阶段

### 第二次错误（13:38）
**错误内容**：
- 提到 "Reviewer 的 MINOR REVISION 意见"（Project 2 的审查结果）
- 提到 "补充 11 条 2023-2025 年最新文献"（Project 2 的修改）
- 提到 "修订后的 PDF"（Project 2 的产出）

**实际情况**：
- Project 3 的 Writer 还在撰写 Results 和 Conclusion
- 还没有进入审查阶段

## 根本原因
1. **上下文污染**：同时处理两个项目，记忆中混杂了两个项目的信息
2. **缺乏项目标识符**：回复时没有明确标注当前项目编号
3. **路径检查不严格**：没有检查文件路径中的项目编号
4. **记忆检索错误**：检索到了 Project 2 的记忆，但误以为是 Project 3

## 用户反馈
- 第一次：指出 "回复错误的串到了 project 2"
- 第二次：再次指出 "回复又串到了 project 2"
- 要求：记录到记忆和 SOUL 中，防止以后再犯

## 改进措施

### 立即措施
1. **每次回复前明确当前项目**：在心里确认 "现在是 Project X"
2. **在回复开头标注项目编号**：如 "## Project 3 当前状态"
3. **检查所有文件路径**：确保路径包含正确的项目编号

### 长期措施
1. **使用项目标识符**：在所有回复中明确标注项目编号
2. **避免跨项目记忆污染**：不要在同一个回复中混用两个项目的状态
3. **定期清理上下文**：完成一个项目后，明确标记该项目已结束
4. **建立检查清单**：每次回复前检查项目编号、文件路径、术语

### 检查清单（每次回复前）
- [ ] 确认当前项目编号（Project 3）
- [ ] 检查所有文件路径是否包含正确的项目编号（paper-project-3）
- [ ] 确认没有混用其他项目的术语（如 "v2"、"MINOR REVISION"）
- [ ] 在回复开头标注项目编号

## 影响
- **严重误导用户**：用户以为 Project 3 已经完成审查，实际还在撰写阶段
- **浪费时间**：用户需要两次指出错误，打断工作流程
- **降低信任度**：连续两次犯同样的错误，严重影响用户对 AI 的信任

## 教训
**铁律：宁可多花 5 秒确认项目，也不要混淆上下文！**

多项目并行时，必须严格区分项目上下文，避免记忆污染。每次回复前，必须明确当前项目编号，检查所有文件路径和术语。

## 预防措施
1. 在 SOUL.md 中添加 "项目上下文管理铁律"
2. 在 MEMORY.md 中记录这次错误
3. 在每次回复前执行检查清单
4. 如果不确定当前项目，先检查文件路径再回复
